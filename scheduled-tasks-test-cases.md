## TC01 · 模板卡片预填触发

**测试目标**：验证点击模板卡片正确预填创建弹窗字段

**初始条件**：
- 用户在初始页面（task-content）
- 已有 4 个模板卡片

**测试步骤**：
1. 点击「每日行业资讯简报」模板卡片 → 创建弹窗打开
2. 检查任务名称 input = "每日行业资讯简报"
3. 检查任务指令 textarea = "搜集 [AI / 科技 / 财经 / 教育] 等行业的最新动态，5条核心新闻"
4. 检查时间模式 Tab = "每天"，时间 = "08:30"，星期 = 周一-周五（active）
5. 检查专家/远控/账号 = 未选（提示用户选）

**预期结果**：所有字段正确预填，无需用户重新输指令；专家等字段允许空白保存

## TC02 · 卡片三点菜单-删除二次确认（关键）

**测试目标**：验证破坏性操作有二次确认

**初始条件**：列表中有 1+ 个任务

**测试步骤**：
1. 点击任务卡片右上角三点
2. 菜单展开（含执行一次/编辑/删除）
3. 点击「删除」
4. 检查出现二次确认弹窗，文案 = "确定删除该定时任务吗？该操作不可恢复"
5. 点击「取消」→ 任务保留
6. 再点删除 → 二次确认 → 点「确认」→ 任务消失

**预期结果**：破坏性操作必经二次确认；取消操作可逆

---

## TC03 · 任务数 50 个上限

**测试目标**：上限不超过 50

**初始条件**：已有 50 个任务

**测试步骤**：
1. 点击「创建任务」→ 弹出创建弹窗
2. 正常填写 + 保存
3. 检查保存行为：弹 Toast「已达最大任务数（50个），请删除部分任务后再创建」

**预期结果**：超限阻止创建；保存按钮置灰或弹 Toast 提示

---

## TC04 · Enter 换行（不绑保存）

**测试目标**：textarea 内 Enter 为换行，不触发保存

**初始条件**：弹窗已打开，焦点在 textarea

**测试步骤**：
1. 在 textarea 输入多行文字
2. 按 Enter → 应该换行（textarea 高度增加）
3. 继续输入
4. 按 Tab 或点别处失焦
5. 弹窗不应该关闭
6. 必须点「创建」/「更新」按钮才保存

**预期结果**：Enter 仅作为换行符；不会误触发保存或关闭弹窗

---

## TC05 · 全不选星期=每天默认时间循环（关键）

**测试目标**：每天模式下不选任何星期也能正常触发

**初始条件**：弹窗时间 Tab = "每天"

**测试步骤**：
1. 选择时间 = 08:30
2. 保持 7 个星期按钮均未 active
3. 点保存
4. 后端创建任务：触发规则 = 每天 08:30（不论星期）
5. 验证：每周一、二、三、四、五、六、日均触发

**预期结果**：全不选星期等价于"每天都执行"，按时间循环

## TC06 · 关闭开关后行为（执行中不被中断）

**测试目标**：关闭开关不影响正在执行的任务

**初始条件**：有一个 1 小时任务正在执行（卡片显示"已触发"或"执行中"），开关当前为开

**测试步骤**：
1. 任务在执行中
2. 用户把开关关闭
3. 检查任务不被中断：继续跑到完成
4. 任务完成后，"上次状态"显示"已执行"
5. 下一次预定时间（如下个 8:30）→ 跳过不触发

**预期结果**：当前执行不被打断；从下次预定触发开始按关闭状态跳过

---

## TC07 · 间隔模式边界值（关键）

**测试目标**：1 / 1440 边界 + 超出校验

**初始条件**：弹窗时间 Tab = "按间隔"

**测试步骤**：
1. 输入 1 分钟 + 起始时间 09:00 → 保存成功
2. 测试 0 分钟 → 输入框禁止输入或提示错误
3. 测试 1441 分钟 → 同上
4. 输入 1440 分钟 + 起始 → 保存成功

**预期结果**：1 ≤ 间隔 ≤ 1440；空值禁用保存；超出弹 Toast/按钮置灰

## TC08 · 单次过去时间算法（关键）

**测试目标**：选定日期=今天、时间已过，自动顺延到明天

**初始条件**：当前时间 2026-07-13 14:00

**测试步骤**：
1. 选日期 2026-07-13、时间 09:00 → 保存
2. 后端触发时间 = 2026-07-14 09:00（即明天）
3. 选日期 2026-07-13、时间 16:00 → 保存
4. 后端触发时间 = 2026-07-13 16:00（即今天，因未过）
5. 选日期 2026-07-15、时间 11:00 → 保存
6. 后端触发时间 = 2026-07-15 11:00（即所选日期，因未来）

**预期结果**：算法遵循 §6.6.2 表格三种情况

## TC09 · 单次 60 天上限

**测试目标**：超过 60 天的日期不可选

**初始条件**：当前日期 2026-07-13

**测试步骤**：
1. date input 打开日期选择器
2. 检查 2026-07-13（今天）可选
3. 检查 2026-09-11（60 天后）刚好可选
4. 检查 2026-09-12（61 天后）不可选
5. 选过去日期 2026-07-10 → 不可选

**预期结果**：仅 [今日, 今日+60] 区间可选

## TC10 · 远控通道非必选（默认发客户端）

**测试目标**：不选远控通道也能保存

**初始条件**：弹窗已打开

**测试步骤**：
1. 不勾选「远控通道」（保持默认 / 留空）
2. 勾选「发送账号 1」
3. 点保存 → 成功
4. 任务执行后，结果通过桌面端 Toast 通知（不进微信/钉钉/飞书）

**预期结果**：保存不阻塞；任务结果通过客户端渠道投递

## TC11 · "执行一次" → "去检查" 跳转 chat（关键）

**测试目标**：执行任务后点击"去检查"进入 chat 对话

**初始条件**：列表中有一个任务

**测试步骤**：
1. 点卡片三点菜单 → 「执行一次」
2. 系统向 chat 发起新对话（含任务指令）
3. 卡片按钮变为「去检查」，状态显示"已触发"
4. 点「去检查」
5. 跳转至 chat.html 该任务对应的新对话页
6. 对话第一条消息/标题包含任务名称 + 任务指令

**预期结果**：执行即发起、不可中断；跳转后能完整看到执行过程

## TC12 · 失败手动重试（"重试"按钮）

**测试目标**：失败后提供手动重试入口

**初始条件**：任务上次执行失败

**测试步骤**：
1. 卡片底部显示「上次：失败（可重试）」
2. 「重试」按钮显示
3. 点「重试」→ 走 TC11 流程
4. 成功后按钮消失，"上次"改为"已执行"
5. 不点「重试」→ 不自动重新触发

**预期结果**：仅手动触发，不自动重试；UI 状态明确

---

## TC13 · 时间冲突-并行执行

**测试目标**：同一时间触发多个任务，按顺序并行启动

**初始条件**：列表中有 3 个任务都设 09:00 触发

**测试步骤**：
1. 系统检测同一时间点
2. 3 个任务同时启动（按创建顺序依次发出执行请求）
3. 每个任务独立在 chat.html 中创建对话
4. 每个对话有独立的 ID

**预期结果**：不串行排队；同时触发，结果都能回客户端

---

## TC14 · 网络异常-本地缓存+联网后执行

**测试目标**：离线时任务不丢，联网后自动执行

**初始条件**：任务预定 09:00 触发；用户在 08:55 关闭网络

**测试步骤**：
1. 09:00 任务到达 → 本地缓存执行内容
2. 卡片显示"待联网执行"
3. 用户 10:00 恢复网络
4. 系统自动重试缓存的任务
5. 任务正常在 chat 中执行 + 创建对话
6. 卡片状态回写"已执行"

**预期结果**：离线不丢任务；联网自动补执行

## TC15 · chat 对话同步（绑定规则）

**测试目标**：执行结果在 chat 中正确归属

**A. 绑定具体专家**（如"AI 助手"）：
1. 任务"每周一行程提醒"，绑 AI 助手
2. 执行后，chat 中 AI 助手会话下新增对话，标题含"每周一行程提醒"

**B. 绑定 main agent**（主专家）：
1. 任务"日报告警"，专家=main（不选具体）
2. 执行后，chat 中 main agent 会话下新增子对话

**预期结果**：对话归属正确，便于用户回溯

## TC18 · 单次任务超时 30 分钟（硬超时）

**测试目标**：超时自动置失败 + 通知

**初始条件**：单次任务已触发，agent 执行中

**测试步骤**：
1. 模拟一个执行时长超过 30 分钟的 agent 任务（mock 服务或后端注入）
2. 在 30:00 后观察：
   - 任务卡片状态自动转为「失败」
   - 弹顶部 Toast「任务执行超时（30 分钟）」
   - 卡片显示「重试」按钮（见 TC12）

**预期结果**：
- 与 §8.4.11 知识库 processing 30 分钟保持一致
- 长任务不应永久挂起
- 用户可手动「重试」（见 TC12）
